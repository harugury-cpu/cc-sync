# skillers-suda

> **4명의 전문가가 수다 떨고 토론하면서 모호한 아이디어를 동작하는 Claude Code 스킬로 바꿔줍니다.**

아이디어 한 마디만 하면 됩니다. 4명의 페르소나 — 기획자/사용자/전문가/검수자 — 가 진짜 병렬 에이전트로 소집되어 의논하고, 구조화된 인터뷰로 안내합니다. 최종 결과는 완전히 스캐폴딩된 스킬/에이전트/커맨드 파일 — 9개 품질 항목 자동 검증, A/B eval 벤치마크, 트리거 최적화까지 완료된 상태로 나옵니다.

## 왜 skillers-suda?

- **코딩 지식 불필요** — 모든 질문에 설명과 장단점이 있어요. 모르겠으면 (추천) 표시된 걸 고르면 됩니다
- **시뮬레이션이 아닌 진짜 에이전트** — 4개의 Claude 서브에이전트가 병렬로 실행되어, 인터뷰가 시작되기 전에 각자의 관점에서 아이디어를 분석합니다
- **멀티 단계 워크플로우 설계** — 단순 프롬프트 스킬이 아닙니다. 6가지 단계 타입(prompt/script/api_mcp/rag/review/generate)이 인터뷰 결과에 따라 자동 조합됩니다
- **내장 품질 검증** — 생성 직후 9개 구조적 항목을 자동 검사하고, FAIL 항목은 결과를 보여주기 전에 자동 수정합니다
- **A/B eval 자동 실행** — 스킬 적용/미적용 결과를 자동 비교해서 실제로 효과가 있는지 확인합니다
- **실제로 작동하는 트리거 최적화** — 과적합 방지를 위한 train/test 분할로 최대 5회 반복하며 description을 정제합니다
- **기존 스킬 분석 모드 포함** — 이미 있는 스킬이나 에이전트 파일을 지정하면 4가지 관점의 검토와 개선 제안을 받을 수 있습니다

## 어떻게 작동하나요

```
유저: "번역 스킬 만들어줘"
         ↓
4명의 전문가 에이전트 병렬 소집 (기획자 / 사용자 / 전문가 / 검수자)
         ↓
"우리끼리 얘기해봤는데, 이렇게 하면 어때요?"
         ↓
구조화된 인터뷰 (3-5개 질문, 각 질문에 선택지 + 설명 포함)
         ↓
워크플로우 설계 (prompt / script / api_mcp / rag / review / generate 단계 조합)
         ↓
SKILL.md + scripts/ + references/ 자동 생성
         ↓
품질 검증 (9개 항목) → FAIL 자동 수정 → 재검증
         ↓
Eval 실행 (with_skill vs. without_skill A/B 비교)
         ↓
Description 최적화 (최대 5회 반복, 60/40 train/test 분할)
         ↓
"테스트해볼까요?" → 피드백 → 고도화 반복
```

## 기능

### 스킬 생성 워크플로우 (9단계)

| 단계 | 내용 |
|------|------|
| A — 아이디어 수집 | AskUserQuestion으로 아이디어를 파악. 대화 컨텍스트에 이미 워크플로우가 있으면 추출 |
| B — 전문가 팀 소집 | 4개 에이전트 병렬 실행. 각 에이전트가 자신의 역할 관점에서 아이디어를 분석 |
| C — 인터뷰 | 3-5개 구조화 질문. 선택지, 설명, 추천 기본값 포함 |
| D — 워크플로우 확인 | 파일 생성 전에 단계 타입과 순서를 확인 |
| E — 파일 생성 | SKILL.md + scripts/ + references/ 스캐폴드 자동 작성 |
| F — Eval | with_skill vs. without_skill 시나리오 비교. 채점 에이전트가 점수 매김. 결과는 eval_review.html로 확인 |
| G — 품질 검증 | verify-skill.py가 9개 구조적 항목 검사. FAIL 자동 수정 후 재검증 |
| H — Description 최적화 | run_loop.py가 트리거/비트리거 쿼리 약 20개 생성. 최대 5회 반복하여 최적 description 선택 |
| I — 테스트 + 고도화 | 대화형 개선 루프 — 톤 조정, API 단계 추가, 스크립트 최적화 |

### 품질 검증 9개 항목

| 항목 | 검증 내용 |
|------|----------|
| frontmatter | YAML 헤더가 올바른 형식인지 |
| name | 스킬 이름이 있는지 |
| description | 트리거 설명이 있는지 |
| third_person | description이 3인칭으로 쓰였는지 |
| trigger_phrases | 트리거 문장이 충분히 있는지 |
| word_count | 내용이 너무 짧지 않은지 |
| imperative_form | 지시문이 명령형으로 쓰였는지 |
| references_exist | references/에 필요한 파일이 있는지 |
| progressive_disclosure | 단계별 공개 구조인지 |

각 항목은 PASS / WARN / FAIL 3단계로 판정됩니다. FAIL 항목은 스킬을 전달하기 전에 자동으로 수정됩니다.

### 워크플로우 단계 타입

| 타입 | 설명 | 예시 |
|------|------|------|
| prompt | Claude가 추론으로 처리 | 텍스트 분석, 요약, 번역 |
| script | 반복성/일관성/API 작업 → Python 또는 Bash | API 호출, 데이터 파싱 |
| api_mcp | 외부 도구 연동 (API 우선, MCP 차선) | Slack 전송, DB 조회 |
| rag | references/에서 지식 검색 | 용어집, 스타일 가이드 |
| review | 품질 검토 (AI 또는 사용자) | 번역 정확도, 코드 품질 |
| generate | 최종 결과물 출력 | 파일 생성, 보고서 |

### 분석 모드

`/skillers-suda 분석 <경로>` 로 기존 스킬이나 에이전트 파일을 분석합니다. 4명의 전문가가 각자의 관점에서 검토하고 통합된 개선 리포트를 작성합니다.

```
/skillers-suda 분석 skills/my-skill/SKILL.md
/skillers-suda 분석 .claude/agents/my-agent.md
```

### 컴포넌트 자동 판단

인터뷰 결과에 따라 사용자의 케이스에 스킬/에이전트/커맨드 중 어떤 형태가 적합한지 자동으로 판단하고 해당 파일 구조를 생성합니다.

## 4명의 전문가

| 전문가 | 역할 | 이런 걸 물어봐요 |
|--------|------|-----------------|
| 기획자 | 방향과 범위 | "누가 쓸 건데? 뭘 해결하는 거야?" |
| 사용자 | UX 검증 | "나라면 이걸 어떻게 쓸까?" |
| 전문가 | 기술적 가능성 | "이 분야는 이런 점을 조심해야 해" |
| 검수자 | 엣지 케이스 탐지 | "이거 이 경우에도 돼?" |

4명 모두 진짜 병렬 Claude 서브에이전트로 소집됩니다 — 역할극 시뮬레이션이 아닙니다.
