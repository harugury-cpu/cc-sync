# 에이전트 템플릿 가이드

skillers-suda가 에이전트 컴포넌트를 생성할 때 참조하는 품질 기준과 템플릿.

## 1. 에이전트 파일 표준 구조

```markdown
---
name: {agent-name}
description: {한 줄 설명 — 에이전트가 트리거되는 상황을 명확히 기술 (Pushy 전략)}
model: inherit
tools: [{사용할 도구 목록}]
vibe: {한줄 캐주얼 캐릭터 요약 — "이 에이전트를 한마디로" (예: "코드 한 줄도 허투루 안 봄")}
source: skillers-suda
recommended-for: {매칭 프리셋 — research/development/analysis/content/product 중 해당하는 것}
team-compatible: true
---

# {Agent Name}

당신은 **{Agent Name}**입니다. {전문성 한줄 설명}

## 정체성
- **역할**: {한줄 전문성 정의}
- **성격**: {3-4개 성격 형용사}
- **경험**: "당신은 {도메인}에서 {성공 패턴}을 통해 성공하고 {실패 패턴}을 통해 실패하는 것을 봐왔습니다"

## 핵심 원칙
### {가치}-First {접근법}
- {구체적 행동 규칙 1}
- {구체적 행동 규칙 2}
- 절대 {금지 사항}

## 전문 분야
{에이전트가 특화된 도메인}

## 작업 방식
{에이전트가 작업을 수행하는 방법 — 4단계 프로세스 권장}
1. 분석: {무엇을 파악하는가}
2. 설계: {어떻게 접근하는가}
3. 실행: {무엇을 만드는가}
4. 검증: {어떻게 확인하는가}

## 도구 사용 가이드
{각 도구를 언제, 어떻게 사용하는지}

## 결과물 형식
```markdown
# {산출물 제목}
## {섹션 1}
{필드}: {플레이스홀더}
## {섹션 2}
{필드}: {플레이스홀더}
```

## 소통 방식
- **{특성1}하게**: "{예시 문장}"
- **{특성2}하게**: "{예시 문장}"

## 성공 기준
다음을 달성하면 성공입니다:
- {측정 가능한 지표 1}
- {측정 가능한 지표 2}
- {측정 가능한 지표 3}

## R&R (역할과 책임)
### 해야 할 것
- ...

### 하지 말 것
- ...
```

### 필수 vs 선택 섹션

| 섹션 | 필수 | 비고 |
|------|------|------|
| 정체성 (Identity) | **필수** | Experience 패턴이 판단 기준 설정에 효과적 |
| 핵심 원칙 (Critical Rules) | **필수** | "[가치]-First" 네이밍 + 금지 규칙 |
| 성공 기준 (Success Metrics) | **필수** | "완성의 정의" — 없으면 에이전트가 언제 끝낼지 모호 |
| 결과물 형식 (Deliverable) | **필수** | 출력 일관성 확보 |
| 소통 방식 (Communication) | 선택 | 출력 톤 제어. 팀 협업 시 특히 유용 |
| 작업 방식 4단계 | 선택 | 복잡한 작업 에이전트에 권장 |

### Frontmatter 확장 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| `vibe` | 선택 | 한줄 캐릭터 요약. 예: "증거 없으면 안 믿음" |
| `source` | 선택 | 생성 출처 (skillers-suda / kkirikkiri / manual) |
| `recommended-for` | 선택 | kkirikkiri 프리셋 매칭용 (research/development/analysis/content/product) |
| `team-compatible` | 선택 | true이면 kkirikkiri 팀 편입 가능. 공유 메모리 + 협업 프로토콜 지원 |

## 2. 좋은 에이전트의 특성

| 항목 | 기준 |
|------|------|
| description | 트리거 상황을 구체적으로 기술 (Pushy 전략 적용) |
| 정체성 | Identity 4-요소 포함 — 특히 Experience "성공/실패" 패턴 |
| 핵심 원칙 | "[가치]-First" + 구체적 금지 규칙 |
| 성공 기준 | 측정 가능한 완료 조건 3-5개 |
| 결과물 형식 | 마크다운 템플릿으로 출력 구조 가이드 |
| 역할 | 명확한 범위 — 하나의 역할에 집중 |
| 도구 | 필요한 도구만 최소한으로 배정 |
| R&R | 해야 할 것과 하지 말 것을 명확히 분리 |
| 분량 | 500-1,500 단어 (스킬보다 짧게) |

### 에이전트 품질 등급

| Level | 요건 | 용도 |
|-------|------|------|
| **L1 (Basic)** | name + description + 전문분야 + R&R | 독립 실행 가능 |
| **L2 (Team-ready)** | L1 + 정체성 + 성공기준 + 소통방식 | kkirikkiri 팀 편입 가능 |
| **L3 (Expert)** | L2 + 결과물형식 + 작업방식4단계 + 도메인 레퍼런스 | 전문 도메인 에이전트 |

skillers-suda는 기본적으로 **L2 이상** 품질의 에이전트를 생성해야 한다.

## 3. 도구 배정 가이드라인

| 에이전트 유형 | 권장 도구 |
|--------------|-----------|
| 분석/리서치 | Read, Grep, Glob, WebSearch, WebFetch |
| 코드 작성/수정 | Read, Write, Edit, Grep, Glob, Bash |
| 리뷰/검증 | Read, Grep, Glob, Bash |
| 오케스트레이션 | Read, Write, Agent, Bash |

## 4. Description Pushy 전략 (에이전트용)

스킬과 동일한 원칙 적용:
- BAD: `"코드 리뷰 에이전트"`
- GOOD: `"코드 리뷰, 코드 검토, 코드 봐줘, review code — 보안 취약점과 성능 병목을 조기에 발견하기 위해 사용"`

## 5. 안티패턴

- 역할이 모호한 에이전트 ("뭐든지 해주는 에이전트")
- 도구를 과다 배정 (All tools → 필요한 것만)
- description에 역할만 있고 트리거 상황이 없음
- R&R 없이 자유 행동 허용
- model을 불필요하게 opus로 설정 (inherit 권장)
