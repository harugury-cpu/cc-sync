# Improvement Principles

> 스킬 반복 개선 시 적용하는 핵심 원칙. Phase G에서 참조.

## 일반화 원칙

스킬 개선 시 특정 테스트 케이스만 통과하도록 하드코딩하지 않는다.

- BAD: eval에서 "Python"이 필요하면 `if "Python" in query` 추가
- GOOD: 프로그래밍 언어 전반을 처리하는 로직으로 일반화

### 규칙

1. **특정 값 하드코딩 금지** — eval 케이스의 구체적 값을 조건문에 넣지 않는다
2. **패턴으로 일반화** — 개별 케이스가 아닌 케이스의 공통 패턴을 해결한다
3. **과적합 탐지** — 새 eval 추가 시 기존 eval이 깨지면 과적합 신호

## Lean 유지

스킬은 가볍게 유지한다. 불필요한 복잡성을 추가하지 않는다.

1. **SKILL.md는 2,000단어 이내** — 넘으면 references로 분리
2. **중복 제거** — 같은 내용이 SKILL.md와 references에 모두 있으면 한쪽 제거
3. **사용하지 않는 코드 제거** — 반복 개선 중 불필요해진 로직은 즉시 삭제

## Why 설명

스킬의 description과 instructions에서 "무엇을 하는가"뿐 아니라 "왜 하는가"를 설명한다.

- BAD: "코드를 리뷰합니다"
- GOOD: "코드를 리뷰합니다 — 보안 취약점과 성능 병목을 조기에 발견하여 프로덕션 장애를 예방하기 위해"

## 반복 코드 번들

반복 개선 시 수정하는 파일들을 번들로 관리한다.

| 번들 | 파일 | 용도 |
|------|------|------|
| Core | SKILL.md | 핵심 워크플로우 |
| Eval | evals/evals.json | 테스트 케이스 |
| Description | SKILL.md frontmatter | 트리거 최적화 |
| References | references/*.md | 보조 지식 |

수정 시 번들 단위로 일관성을 유지한다. Core를 바꾸면 Eval도 함께 업데이트.
