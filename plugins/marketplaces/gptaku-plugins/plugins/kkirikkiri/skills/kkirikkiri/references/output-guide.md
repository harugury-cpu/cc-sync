# 결과 전달 + 저장 가이드

> **Step 8에서 결과 전달, 팀 저장, 에이전트 저장 시 반드시 참조한다.**

---

## Auto-memory 저장 유도 (Step 8-2)

결과 전달 후 팀 운영 맥락을 아래 형식으로 자연어 요약 출력한다.
단순 설정이 아닌 **무엇이 효과적이었고 왜**가 핵심.

```
끼리끼리 세션 요약:

환경: Codex CLI ✓, Gemini CLI ✓, 패키지 매니저 bun

팀 구성과 결과:
- 프리셋: research, 4명 (팀장 + 리서처 2 + 검증자 1)
- 효과적이었던 점: [예: 검증자가 리서처의 편향을 잡아냄]
- 비효과적이었던 점: [예: 리서처 2명이 같은 소스를 중복 조사]
- 다음에 개선할 점: [예: 리서처에게 소스 범위를 명시적으로 분할]

공유 메모리 위치:
- {KKIRIKKIRI_DIR}/TEAM_PLAN.md — 계획
- {KKIRIKKIRI_DIR}/TEAM_FINDINGS.md — 결과 (DEAD_ENDS 섹션 포함)
```

이 요약이 Auto-memory에 저장되면 다음 세션에서:
- 환경 스캔 축소, 팀 구성 추천에 활용
- 교체 팀원에게 공유 메모리 인덱스로 즉시 온보딩 가능
- 이전에 실패한 전략을 반복하지 않음

## 팀 저장 파일 형식 (Step 8-3)

저장 경로: `{프로젝트루트}/.kkirikkiri/shared/saved-teams/{team_name}.md`

```markdown
# 저장된 팀: [팀 이름]

- 생성일: [날짜]
- 프리셋: [preset]
- 목표: [인터뷰에서 파악한 목표]

## 팀 구성
| 역할 | 모델 | 담당 |
|------|------|------|
| [팀장] | Opus | [업무] |
| [팀원1] | [모델] | [업무] |

## 인터뷰 답변 요약
- Q1: [답변]
- Q2: [답변]

## 환경 조건
- Codex CLI: [있음/없음]
- Gemini CLI: [있음/없음]

## 성과
- 라운드: [수행 라운드 수]
- 품질 판정: [결과]
```

### 저장된 팀 불러오기

사용자가 "저번 리서치 팀 다시 써줘" 유사 요청 시:
1. `.kkirikkiri/shared/saved-teams/` 디렉토리 스캔
2. 관련 팀 파일 Read로 확인
3. 저장된 구성 기반으로 Step 4로 직행 (인터뷰 생략)
4. 환경이 달라졌으면 자동 조정

```
"저번에 사용한 '[팀 이름]' 팀을 찾았어요. 같은 구성으로 시작할까요?"
```

## 팀원 에이전트 저장 절차 (Step 8-3-1)

### 저장 처리 순서

**1. 저장할 팀원 선택**
- 팀장 제외, 성과가 좋았던 팀원 추천
- 여러 명이면 AskUserQuestion으로 multiSelect 확인

**2. 프롬프트 정제**
- 제거: 공유 메모리 경로, 팀 이름, 날짜/시간, 다른 팀원 이름/역할 참조
- 유지: 핵심 역할 설명, 전문 분야, 작업 방식, 도구 사용 가이드, R&R

**3. 에이전트 파일 생성** — `.claude/agents/{역할명}.md`

```markdown
---
name: {역할명}
description: {역할 설명 — Pushy 전략 적용}
model: inherit
tools: [{팀원이 사용한 도구 목록}]
source: kkirikkiri
created: {날짜}
recommended-for: {매칭된 프리셋 이름}
---

# {역할명}

{정제된 프롬프트 내용}
```

**4. 이름 충돌 처리**

같은 이름의 에이전트가 이미 있으면:
```
"이미 같은 이름의 전문가가 있어요. 어떻게 할까요?"
- 덮어쓰기
- 다른 이름으로 저장
- 건너뛰기
```

**5. 저장 완료 안내**
```
"[역할명] 전문가를 저장했어요! 다음에 팀을 만들 때 자동으로 감지될 거예요."
```
