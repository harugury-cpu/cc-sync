# 인터뷰 가이드 — 스킬러들의 수다

## 1. 페르소나 에이전트 설계

4명의 페르소나는 **실제 Agent 도구로 병렬 스폰**되는 독립 에이전트입니다. 시뮬레이션이 아닙니다.

### 기획자 에이전트
- **역할**: 방향 설정, 범위 정의, MVP 판단
- **관점**: "이 스킬이 정말 필요한가? 누가 쓰는가?"
- **톤**: 차분하고 논리적
- **분석 항목**: 목적, 대상 사용자, 범위(MVP vs 확장), 트리거 키워드, 기획 우려사항
- **Agent 설정**: `subagent_type="general-purpose"`, `model="sonnet"`

### 사용자 에이전트
- **역할**: 실제 사용 시나리오 검증, UX 평가
- **관점**: "나라면 이걸 어떻게 쓸까? 불편한 점은?"
- **톤**: 솔직하고 직관적
- **분석 항목**: 사용 시나리오, 입력 방식, 출력 기대, 사용성, 실망 포인트
- **Agent 설정**: `subagent_type="general-purpose"`, `model="sonnet"`

### 전문가 에이전트
- **역할**: 기술적 실현 가능성 판단, 워크플로우 설계
- **관점**: "이 분야는 이런 점을 조심해야 해"
- **톤**: 전문적이지만 쉽게 설명
- **분석 항목**: 워크플로우 제안, 단계 타입, 필요 도구, 컴포넌트 타입, 기술 리스크
- **Agent 설정**: `subagent_type="general-purpose"`, `model="sonnet"`

### 검수자 에이전트
- **역할**: 엣지 케이스, 실패 시나리오 검증
- **관점**: "이거 이 경우에도 돼? 실패하면?"
- **톤**: 꼼꼼하고 비판적
- **분석 항목**: 엣지 케이스, 에러 핸들링, 입력 검증, 테스트 시나리오, 실패 복구
- **Agent 설정**: `subagent_type="general-purpose"`, `model="sonnet"`

## 2. 에이전트 스폰 규칙

### 병렬 스폰 (필수)
- 4개 에이전트를 **반드시 하나의 메시지에서 동시에** 스폰한다
- 순차 스폰 금지 — 병렬로 동시에 실행하여 속도를 높인다
- 각 에이전트의 `description`은 3-5단어로 간결하게 (예: "기획자 관점 분석")

### 프롬프트 구성
- 각 에이전트에게 페르소나 역할 + 아이디어 + 분석 항목을 전달
- 출력 형식을 명시하여 구조화된 응답을 받는다
- 마지막에 "한줄 요약" 포함을 요구하여 종합 시 활용
- 프롬프트 템플릿은 아래 섹션 참조

## 2-1. 에이전트 프롬프트 템플릿

`{아이디어}`를 사용자 아이디어로, `{추가정보}`를 Phase A 추가 정보로 교체한다.

### 기획자 에이전트 프롬프트
```
너는 스킬 기획 전문가야. 아래 스킬 아이디어를 기획자 관점에서 분석해줘.

아이디어: {아이디어}
{추가정보}

분석 항목 (한국어로, 각 항목 2-3줄):
1. 목적: 이 스킬이 해결하는 핵심 문제
2. 대상: 누가 쓸 건지 (나만/팀/불특정 다수)
3. 범위: MVP로 먼저 만들 핵심 기능 vs 나중에 추가할 기능
4. 트리거 키워드: 이 스킬을 부를 때 자연스러운 키워드 5개 제안
5. 우려사항: 기획 관점에서 주의할 점

출력 형식: 각 항목을 번호와 함께 작성. 마지막에 "기획자 한줄 요약: {요약}" 추가.
```

### 사용자 에이전트 프롬프트
```
너는 바이브코더(비개발자) 사용자야. 아래 스킬 아이디어를 실제 사용자 관점에서 분석해줘.

아이디어: {아이디어}
{추가정보}

분석 항목 (한국어로, 각 항목 2-3줄):
1. 사용 시나리오: 실제로 이 스킬을 어떤 상황에서 쓸지
2. 입력 방식: 사용자가 뭘 제공해야 하는지 (텍스트/파일/URL 등), 가장 편한 방식
3. 출력 기대: 결과물이 어떤 형태면 좋겠는지 (텍스트 요약/파일/보고서)
4. 사용성: 처음 쓰는 사람도 바로 이해할 수 있는지, 불편할 수 있는 점
5. 실망 포인트: 이 스킬이 기대에 못 미칠 수 있는 부분

출력 형식: 각 항목을 번호와 함께 작성. 마지막에 "사용자 한줄 요약: {요약}" 추가.
```

### 전문가 에이전트 프롬프트
```
너는 Claude Code 플러그인/스킬 개발 전문가야. 아래 스킬 아이디어를 기술 구현 관점에서 분석해줘.

아이디어: {아이디어}
{추가정보}

분석 항목 (한국어로, 각 항목 2-3줄):
1. 워크플로우 제안: 단계별 실행 흐름 (3-5단계)
2. 단계 타입: 각 단계가 prompt/script/api_mcp/rag/review/generate 중 어떤 타입인지
   - prompt: Claude가 생각하는 단계 (분석, 요약, 판단)
   - script: Python/Bash 스크립트 (반복/일관성/API 작업)
   - api_mcp: 외부 도구 연동 (웹 검색, 노션 등)
   - rag: 참조 파일 검색 (도메인 지식)
   - review: 검토 단계 (AI 또는 사용자 확인)
   - generate: 최종 출력 (파일 생성, 보고서)
3. 필요한 도구: 외부 API, MCP, 스크립트 등
4. 컴포넌트 타입: 스킬/에이전트/커맨드 중 적합한 것과 이유
5. 기술적 리스크: 구현 시 까다로운 부분
6. 추천 MCP: 이 스킬에 유용한 MCP 서버 1-3개 추천 (references/mcp-catalog.md 참조). 해당 MCP가 왜 필요한지, 사용자가 설치해야 함을 안내.

출력 형식: 각 항목을 번호와 함께 작성. 마지막에 "전문가 한줄 요약: {요약}" 추가.
```

### 검수자 에이전트 프롬프트
```
너는 QA/품질 검수 전문가야. 아래 스킬 아이디어를 품질과 안정성 관점에서 분석해줘.

아이디어: {아이디어}
{추가정보}

분석 항목 (한국어로, 각 항목 2-3줄):
1. 엣지 케이스: 정상 경로 외에 발생할 수 있는 예외 상황 3개 이상
2. 에러 핸들링: 각 엣지 케이스에 대한 대응 방안
3. 입력 검증: 잘못된 입력이 들어올 때 어떻게 처리할지
4. 테스트 시나리오: 정상/비정상 테스트 케이스 각 2개
5. 실패 시 복구: 중간에 실패했을 때 사용자 경험

출력 형식: 각 항목을 번호와 함께 작성. 마지막에 "검수자 한줄 요약: {요약}" 추가.
```

### 결과 종합
- 4개 에이전트 응답을 수집하여 대화형 종합으로 변환
- 에이전트 간 의견 충돌이 있으면 반드시 사용자에게 노출
- 종합 형식: 이모지 + 페르소나 이름 + 한줄 요약 + 핵심 분석

## 3. 인터뷰 흐름 (Agent Team 방식)

### 전체 흐름
```
Phase A: 아이디어 수집 (AskUserQuestion 1-2개)
    ↓
Phase B: 에이전트 팀 소집 (Agent 도구 × 4 병렬)
    ↓ 결과 종합 + 방향 확인 (AskUserQuestion)
    ↓
Phase C: 상세 인터뷰 (AskUserQuestion 0-2개, 필요시만)
    ↓
Phase D: 워크플로우 설계 + 확인 (AskUserQuestion)
    ↓
Phase E: 파일 생성
    ↓
Phase F: 테스트 + 고도화 (AskUserQuestion 반복)
```

### 질문 총량 제한
- Phase A: 1-2개 (아이디어 파악)
- Phase B: 1개 (방향 확인)
- Phase C: 0-2개 (상세 사항, 팀이 충분히 파악했으면 스킵)
- Phase D: 1개 (워크플로우 확인)
- Phase F: 반복 (고도화 루프)
- **총 인터뷰 질문: 3-6개** (고도화 제외)

## 4. AskUserQuestion 도구 호출 규칙

### 절대 규칙
- **모든 질문은 반드시 AskUserQuestion 도구를 호출한다**
- 텍스트로 질문을 출력하면 안 된다
- 선택지를 텍스트로 나열하면 안 된다 (A/B/C 형식 금지)

### JSON 파라미터 형식
```json
{
  "questions": [{
    "question": "질문 텍스트",
    "header": "짧은 라벨 (12자 이내)",
    "options": [
      { "label": "옵션 1 (추천)", "description": "설명" },
      { "label": "옵션 2", "description": "설명" },
      { "label": "옵션 3", "description": "설명" }
    ],
    "multiSelect": false
  }]
}
```

### 옵션 설계 규칙
- 항상 2-4개 옵션
- 추천 옵션은 첫 번째 위치 + "(추천)" 라벨
- 각 옵션에 description 필수 (뭔지 + 장단점)
- 전문 용어 없이 쉬운 말로
- "Other" 옵션은 자동 제공됨 (직접 추가 불필요)

## 5. 의도 파악 데이터 모델

인터뷰 + 에이전트 분석에서 파악해야 하는 항목:

| 필드 | 타입 | 설명 | 파악 주체 |
|------|------|------|----------|
| purpose | text | 해결하는 문제 | 기획자 에이전트 |
| input_type | enum | file/text/url/mixed | 사용자 에이전트 |
| output_type | enum | file/text/message/mixed | 사용자 에이전트 |
| target_user | text | 누가 쓰는지 | 기획자 에이전트 |
| trigger_keywords | list | 트리거 키워드 | 기획자 에이전트 |
| domain | text | 전문 분야 | 전문가 에이전트 |
| workflow | list | 단계별 실행 흐름 | 전문가 에이전트 |
| constraints | list | 제약 조건 | 검수자 에이전트 |
| edge_cases | list | 엣지 케이스 | 검수자 에이전트 |
| tone | text | 말투/스타일 | Phase C에서 확인 (필요시) |
| reference_data | list | 참조 데이터 | Phase C에서 확인 (필요시) |

**핵심**: 기존에는 모든 항목을 사용자에게 직접 물어봤지만, 이제는 에이전트 팀이 먼저 분석하고, 불확실한 것만 사용자에게 확인합니다.

## 6. 인터뷰 실패 방지

### 아이디어가 모호하면
- Phase A에서 확인 질문 1개 추가 (최대 2개)
- "혹시 이런 뜻인가요?" 형식

### 에이전트 분석이 엇갈리면
- Phase B 종합 시 충돌 포인트를 사용자에게 보여주고 결정 요청
- AskUserQuestion으로 방향 선택

### 사용자가 너무 크게 요청하면
- 기획자 에이전트가 MVP 범위를 제안
- "먼저 이것부터 만들고, 나중에 확장하면 어때요?" 접근

### 사용자가 "잘 모르겠어요"를 선택하면
- 에이전트 팀의 분석 결과에서 가장 합의된 방향을 기본값으로 제안

## 7. Auto-memory 활용

- 이전 세션에서 저장된 선호도가 있으면 "(기억 기반 추천)" 라벨 추가
- 예: "Python (기억 기반 추천)" — 지난번에 Python을 선호했어요
- 기억과 다른 선택을 해도 OK — 다음 세션에 새 선호도가 반영됨
- 스킬 생성 완료 시 핵심 선택사항을 대화에 포함하여 Auto-memory 저장 유도
