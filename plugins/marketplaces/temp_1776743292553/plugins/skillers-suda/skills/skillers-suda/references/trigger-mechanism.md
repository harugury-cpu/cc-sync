# Trigger Mechanism

> 스킬이 Claude Code에서 트리거되는 메커니즘과 최적화 전략. Phase H에서 참조.

## 트리거 방식

Claude Code는 사용자 메시지를 받으면 설치된 모든 스킬의 description을 스캔한다.
description이 사용자 의도와 매치되면 해당 스킬이 트리거된다.

```
사용자 메시지 → description 매칭 → SKILL.md 로드 → 실행
```

## Undertrigger 경향

스킬은 기본적으로 **트리거가 안 되는 쪽으로 편향**된다.

- 사용자가 정확한 키워드를 쓰지 않으면 트리거 안 됨
- 유사한 표현이나 우회적 요청은 놓침
- 특히 한국어/영어 혼용 시 매칭 실패 빈도 높음

## Pushy Description 전략

undertrigger를 방지하기 위해 description을 적극적으로 작성한다.

### 원칙

1. **넓은 트리거 범위** — 핵심 키워드뿐 아니라 유사 표현, 동의어, 관련 행동 포함
2. **구체적 시나리오 나열** — "코드 리뷰"뿐 아니라 "코드 봐줘", "이거 괜찮아?", "버그 있나?" 등
3. **한/영 혼합** — 한국어와 영어 트리거를 모두 포함
4. **행동 동사 중심** — 명사보다 동사로 의도를 매칭

### 예시

```
BAD:  "코드 리뷰 도구"
GOOD: "코드 리뷰, 코드 검토, 코드 봐줘, review my code, check this code, 버그 찾아줘, 이 코드 괜찮아?"
```

## 복잡한 쿼리만 트리거

단순 질문에는 트리거하지 않고, 스킬이 진정으로 필요한 복잡한 요청에만 반응하도록 설계한다.

- 단순: "Python이 뭐야?" → 트리거 X
- 복잡: "Python으로 웹 크롤러 만들어줘" → 트리거 O

description에서 복잡성 신호를 포함:
- "만들어", "생성해", "설계해", "분석해", "최적화해" 등 **작업 동사**
- "프로젝트", "시스템", "워크플로우" 등 **규모 명사**

## should-trigger / should-not-trigger

eval 설계 시 트리거 정확도도 테스트한다.

| 유형 | 설명 | 예시 |
|------|------|------|
| should-trigger | 스킬이 반드시 트리거되어야 하는 쿼리 | "이 코드 리뷰해줘" |
| should-not-trigger | 스킬이 트리거되면 안 되는 쿼리 | "Python 문법 알려줘" |

run_loop.py에서 이 구분을 사용하여 description의 트리거 정확도를 측정한다.
