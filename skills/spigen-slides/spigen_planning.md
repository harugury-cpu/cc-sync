# spigen_planning.md — 기획·구성 계획 워크플로우
> Step 1 (기획) + Step 2 (구성 계획). 전체 PPT 제작 시 참조.

## Step 1: 기획 (slides-grab 방식)

### 1-0. 요청 유형 감지 (질문 전 분기)

Q1 답변 또는 최초 요청 문구에서 아래 키워드가 감지되면 **1-1 질문 플로우를 건너뛰고** 즉시 자료 수집 모드로 진입한다.

감지 조건:
- KPI / 성과 / 실적 / 달성률 / 목표 보고 / 상반기 보고 / 연간 보고 / H1 / H2

**→ 자료 수집 모드 진입:**

```
"KPI·성과 보고서군요. 관련 자료를 공유해주시면 내용을 채워서 드래프트를 만들어드릴게요.
아래 중 편한 방식으로 주세요:

① Google Sheets / Docs 링크
② 기존 PPT / 보고서 링크 또는 캡처
③ 텍스트로 직접 붙여넣기"
```

자료 수신 후 Claude가 수행할 작업:
1. 자료에서 구조화 데이터 추출:
   - KPI 항목명 / 목표 / 실적 / 달성률 / 가중치
   - 세부 내용 (무엇을 했는가)
   - 계획 (다음 단계)
   - 측정산식 / 정의 / 증빙 (있으면)
2. 자동 결정 (묻지 않음):
   - 테마: **light 고정 — dark 선택 불가**
   - 용도: 디테일용 고정
   - 컴포넌트: `mk_kpi_status_light()` + `mk_kpi_dense_table()` 세트 고정
3. 드래프트 아웃라인 제시 (1-4 형식으로)
4. 사용자 승인 후 실행

**자료에서 값을 읽을 수 없는 필드만 질문한다.** 전부 비어 있으면 그 항목만 텍스트로 물어본다.

---

### 1-1. 준비 질문 수집 (일반 보고서용)

> KPI·성과 보고 유형이 아닌 경우에만 아래 질문 플로우 진행.

질문을 **하나씩 순서대로** 제시한다. 각 질문에 선택지를 함께 제공한다.
"Decide for me" 선택 시 Claude가 맥락에 맞는 최적 옵션을 결정하고 이유를 한 줄 설명한다.

---

**Q1. 주제**
"어떤 내용의 프레젠테이션인가요?" (자유 입력)

**Q2. 청중**
"누가 보는 자료인가요?"
① 내부 팀 (실무 동료)
② 경영진 (의사결정권자)
③ 외부 고객 / 협력사
④ 일반 대중 / 커뮤니티
⑤ 기타 (직접 입력)

**Q2-1. 자료 용도**
"이 슬라이드를 어떻게 사용할 건가요?"
① 축약결론용 — 발표자가 직접 설명, 슬라이드는 핵심만 보조
② 디테일용 — 슬라이드 자체로 이해, 세부 내용이 한 번에 담겨야 함

디테일용 선택 시 내용 규칙:
- 슬라이드당 본문 글머리 3줄 → 5~6줄까지 허용
- 수치·근거·세부 단계 생략 없이 전부 기입
- 카드·블록 설명이 자기완결적이어야 함 (발표자 설명 없이 이해 가능)

**Q3. 발표 길이** → 내지 장수 자동 결정
"발표 시간이 얼마나 되나요?"
① 5분 이하 (라이트닝톡) → 내지 2~3장
② 10분 → 내지 3~4장
③ 15~20분 → 내지 5~6장 (섹션 구분 가능)
④ 30분 이상 → 내지 7장+ (섹션 구분 + 목차)
⑤ 발표 없음 (읽기용 문서) → 내지 장수 제한 없음 (내용 기준)

**Q4. 덱의 목적**
"이 슬라이드로 무엇을 달성하고 싶으신가요?"
① 동기부여 (청중에게 확신 심어주기)
② 사용법 전수 (구체적 실습·방법론)
③ 조직 설득 (도입 제안·승인 요청)
④ 홍보·소개 (커뮤니티·외부 공개)
⑤ 기타 (직접 입력)

**Q5. 테마**
> KPI·성과 보고 유형이면 이 질문을 건너뛴다 — **light 고정, 사용자에게 묻지 않음.**

"어떤 테마로 만들까요?"
① Dark (기본 / 권장)
② Light

**Q6. 방식** (내부 분기 — 선택지 없이 Claude가 판단)
제안서·시안·Bot 소개류 → 콘텐츠 템플릿 방식
보고서·분석·내부 공유 → spigen_lib 방식

---

자동 처리 (묻지 않음):
- **톤 앤 매너**: 항상 ③ 실용적·정보 중심 (팁팩·요점 위주) 고정
- **슬라이드 제목 스타일**: 항상 ① 명사형 고정
- **날짜**: 언급 없으면 오늘 날짜
- **버전**: 언급 없으면 V1.0
- **closing(요약 페이지)**: 항상 생략 (요약 없음 고정)

### 1-2. 비주얼 테제 작성

수집한 정보를 바탕으로 **한 문장**으로 무드·방향·에너지를 정의한다.

예시: _"제품의 신뢰성을 데이터로 증명하는, 간결하고 자신감 있는 톤의 덱"_

### 1-3. 내러티브 구조 설계

**⚠️ 내지 장수: KPI 보고서·임원보고용(경영진 청중) 외에는 장수 제한 없음. 내용 기준으로 결정한다.**
cover·closing·toc·section-divider는 카운트에서 제외한다.

Q3 발표 길이에 따라 내지 장수를 결정한다:

| Q3 선택 | 내지 장수 | 섹션 구분 | 목차 |
|--------|---------|---------|-----|
| ① 5분 이하 | 2~3장 | 생략 | 생략 |
| ② 10분 | 3~4장 | 4장일 때만 | 생략 |
| ③ 15~20분 | 5~6장 | 포함 | 섹션 2개 이상이면 추가 |
| ④ 30분 이상 | 제한 없음 (내용 기준) | 포함 | 포함 |
| ⑤ 발표 없음 | 내용 기준 | 생략 | 생략 |

slides-grab의 내러티브 시퀀스를 아래 단계로 구성한다 (발표용은 장수 제한 없음):

| 단계 | 역할 | 권장 컴포넌트 | 내지 장수 |
|-----|-----|------------|---------|
| **Opener** | 전제·약속·정체성 제시 | cover (고정) | 카운트 제외 |
| **Problem / Context** | 현황·문제·배경 | `mk_3col()` 또는 `mk_split()` | 1장 |
| **Solution / Proof** | 핵심 근거·메커니즘·데이터 | `mk_flow()` 또는 `mk_3col_cards()` | 1~2장 |
| **Impact / Detail** | 구체적 결과·수치·기대효과 | `mk_text_block()` 또는 `mk_split_cards()` | 0~1장 |
| **Close / CTA** | 결론·권고·다음 액션 | closing (선택) | 카운트 제외 |

섹션 구분(section-divider)은 내지 4장 이상일 때만 사용한다. 3장 이하면 생략.
섹션 구분은 **전체 덱에 2개 이상 있을 때만** 사용한다. 섹션 구분이 1개뿐이라면 전체 생략하고 내용 슬라이드로 바로 진입한다.

**closing(요약 페이지):** 항상 생략. 내지 마지막 장으로 마무리.

### 1-4. 아웃라인 상세 제시

각 슬라이드의 **실제 들어갈 텍스트·구조**까지 구체적으로 제시한다.  
사용자가 승인 전에 "이 내용이 이 슬라이드에 이렇게 들어가는구나"를 바로 알 수 있어야 한다.

출력 형식 — 슬라이드마다 아래 항목을 전부 작성한다:

```
[슬라이드 N] (유형) — 컴포넌트: mk_xxx()
컴포넌트 선택 이유: [데이터 유형] | [항목 수: N개] | [구조: 비교/프로세스/독립항목] | [결정 근거]
핵심 메시지: "___는 ___하다"
제목: (슬라이드에 표시될 실제 제목 텍스트)
구성:
  - (블록/카드 이름): (들어갈 텍스트 또는 항목 목록)
  - (블록/카드 이름): (들어갈 텍스트 또는 항목 목록)
  ...
```

> `select_component()`를 호출해 반환된 rationale 값을 "컴포넌트 선택 이유"에 그대로 기입한다.
> 이 값이 기획자 서브에이전트(COMPONENT_BRIEF)로 전달되어 생성 후 검수에 활용된다.
> 문장으로만 적지 말고, 실행 단계에 그대로 넘길 수 있도록 슬라이드별 `COMPONENT_BRIEF`를 구조화해 보관한다.

권장 보관 형식:

```json
{
  "slide_no": 2,
  "component_name": "3col-cards",
  "function_name": "mk_3col_cards",
  "rationale": "독립 항목 3개 카드",
  "inputs": {
    "content_type": "independent_items",
    "item_count": 3,
    "has_comparison": false,
    "is_process": false,
    "has_status": false,
    "is_bilateral": false,
    "is_kpi": false
  }
}
```

실행 단계 규칙:

```txt
- Step 1에서 확정된 function_name 외 다른 컴포넌트로 임의 변경 금지
- 분량 초과/오버플로가 나면:
  1) 텍스트 압축
  2) 같은 컴포넌트 안에서 줄 수 축소
  3) 그래도 안 되면 슬라이드 분리
- "렌더링이 어려워 보여서" 다른 컴포넌트로 바꾸는 것은 금지
```

예시:

```
[슬라이드 1] (cover) — 템플릿 복사
제목: 2026년 1분기 KPI 보고
(부제·담당자·날짜·버전은 자동 처리)

[슬라이드 2] (내지 1) — 컴포넌트: mk_3col_cards()
컴포넌트 선택 이유: 독립 항목 3개 병렬 | 항목 수: 3개 | 구조: 독립항목 | 동등·독립·각각 강조 대상
핵심 메시지: "3개 핵심과제 모두 진행 중이다"
제목: 3대 핵심과제 현황
구성:
  - 카드 A / 온라인 패키지 리뉴얼: 방향성 수립 중 · 2/5 단계 / A안 바코드 스티커 리뉴얼 / B안 패키지 오픈방식 변경 / C안 친환경 펄프몰드
  - 카드 B / 케이스 매뉴얼 프로세스: 전 단계 완료 / 이력 정리·매뉴얼 기준 협의·API 자동화·파트 내 교육 완료
  - 카드 C / 폴더블 매뉴얼 QR화: SQ 검토 결과 대기 / 부문장 보고 완료·삽지 디자인 완료 (4/24)
```

내지 4장 필요 시 슬라이드 추가, closing은 마지막으로 밀린다.

**사용자에게 이 상세 아웃라인을 보여주고 승인받는다. 승인 전 Step 2로 넘어가지 않는다.**  
수정 요청이 있으면 해당 슬라이드의 텍스트·구성만 수정하고 다시 제시한다.

## Step 2: 슬라이드 구성 계획 (디자인 원칙 적용)

승인된 아웃라인을 **공통 SlideSpec / ComponentSpec**으로 정리하고, 그 spec을 기준으로 preview HTML과 Google Slides 렌더링을 모두 생성한다.

### 레이아웃 선택 규칙

HTML은 preview renderer일 뿐 source of truth가 아니다.  
source of truth는 슬라이드별 `SlideSpec + ComponentSpec`이며, HTML과 Google Slides는 같은 spec을 소비해야 한다.

| 슬라이드 유형 | slides-grab 레이아웃 | 비고 |
|------------|-------------------|-----|
| 표지 (Cover) | **Spigen 브랜드 HTML** (Step 3-4에서 교체) | 담당자/날짜/버전 포함 |
| 목차 (Contents) | `contents` | 섹션 2개 이상일 때 추가 |
| 섹션 구분 | `section-divider` | 포스터 스타일, 최소 텍스트 |
| 일반 내용 | `content` | 설명·근거·스토리 |
| 3열 항목 | `statistics` 또는 `split-layout` | 3개 병렬 항목 나열 시 |
| 인용·강조 | `quote` | 임팩트 있는 한 줄 메시지 |
| 변경이력 | `content` | 변경이력 있을 때만 추가 |
| 마지막 | `closing` | 요약 필요 시만 추가 (사용자 확인 후) |

### 컴포넌트 선택 전 필수 판단

컴포넌트를 선택하기 전에 `spigen_render_rules.md` 섹션 8의 **양식 선택 전 필수 판단 기준 Q1~Q3**을 먼저 적용한다.  
항목이 3개라는 이유만으로 형태를 먼저 선택하지 않는다.

**내용 우선 원칙 (레이아웃 선택 순서)**
1. 내용의 항목 수·줄 수·텍스트 길이를 먼저 파악한다
2. 그 내용을 온전히 담을 수 있는 컴포넌트를 선택한다
3. 내용이 컴포넌트 한도를 초과하면 → 더 큰 컴포넌트로 교체하거나 페이지를 분리한다

내용을 줄이거나 잘라서 레이아웃에 끼워맞추지 않는다.
레이아웃이 내용에 맞춰야 하며, 내용이 레이아웃에 맞춰 잘려선 안 된다.

**구성 우선 원칙**
컴포넌트를 먼저 고르지 않는다. 슬라이드가 전달해야 할 것을 먼저 정한 뒤 컴포넌트를 선택한다.
카드는 기본값이 아니다. 카드는 구조와 이해를 실질적으로 돕는 경우에만 사용한다.
기본 레이아웃은 텍스트 블록과 강한 여백이다. 카드·구분선·장식 요소는 필요가 증명될 때만 추가한다.

→ 컴포넌트 빠른 선택 가이드: `spigen_render_rules.md` 섹션 8 참조

### 카피 작성 원칙 (slides-grab 기준)

- **슬라이드당 하나의 메시지**: 경쟁하는 블록 금지
- **3~5초 스캔 가능하게**: 본문 글머리는 3줄 이내
- **제목 스타일**: 자동 처리 — 항상 명사형 ('AI 활용 원칙', '비용 절감 사례')
- **섹션 구분 슬라이드**: 번호 + 짧은 제목만, 설명 없음
- **여백 우선**: 공간을 채우려 하지 말 것
- **cover·section-divider는 포스터처럼**: 하나의 강한 메시지만. 통계·목록·설명을 넣지 않는다. 대시보드가 아니다
- **AI 클리셰 금지**: 혁신적인·원활한·극대화·시너지·솔루션·최적화·스마트한·강력한·게임체인저 사용 금지 → 구체적 수치·사실로 대체
- **코드·함수명 금지**: 슬라이드 텍스트에 `mk_flow()`, `spigen_lib.py`, `slide_base()` 같은 함수명·파일명·변수명 직접 기재 금지 — 기능이나 결과를 일반 언어로 풀어 쓴다
- **불필요한 선·여백 금지**: 장식용 구분선, 빈 사각형, 투명 여백 도형 추가 금지. 내용이 적으면 레이아웃을 단순화한다 — 도형으로 채우지 않는다
- **제거 가능성 검토**: 모든 텍스트·뱃지·콜아웃에 대해 "이것이 없어도 의미가 전달되는가?"를 묻는다. 그렇다면 삭제한다

### 덱 내러티브 리듬

슬라이드 순서 설계 시 아래 리듬 원칙을 따른다.

- **오프너 → 근거 → 상세 → 결론/CTA**: 덱 전체를 이 흐름으로 구성한다. 순서가 흐트러지면 청중은 맥락 없이 정보를 받는다
- **밀도 교차**: 정보 밀도 높은 슬라이드(statistics, content, split) 뒤에는 시각 템포를 리셋하는 단순 슬라이드(section-divider, quote)를 배치한다
- **섹션 구분이 템포 리셋**: section-divider는 새 섹션의 시작점이자 청중에게 숨 쉴 공간을 준다. 섹션 구분 없이 밀도 높은 슬라이드를 연속 3장 이상 쌓지 않는다
- **도입부는 단순하게**: cover는 경쟁하는 정보 없이 제목·전제·약속 하나만 담는다

### 청중별 카피 규칙 (Q2 선택 반영)

Q2에서 선택한 청중에 따라 제목 스타일·본문 밀도·용어 기준을 아래 규칙으로 조정한다.

**① 내부 팀 (실무 동료)**
- 제목: 명사형 또는 서술형 모두 허용 — 팀 내 통용 약어·용어 그대로 사용
- 본문: 현황·프로세스·수치 중심, 3줄 이내 밀도 허용
- 용어: 내부 프로젝트명·코드명·직책 사용 가능
- 톤: 간결·건조, 배경 설명 최소화

**② 경영진 (의사결정권자)**
- 제목: 명사형 고정 (자동 처리)
- 본문: 수치·임팩트·결론 먼저, 세부 내용도 슬라이드에 전부 기입 (디테일용 원칙)
- 용어: 내부 용어 최소화, 비즈니스 성과 언어로 치환
- 톤: 간결·자신감, "~합니다" 확정형 선호
- **섹션 구분 없음**: 임원보고용은 section-divider를 사용하지 않는다
- **페이지 압축 우선**: 4장 이내에 최대한 밀도 있게 합쳐서 끝낸다. 4장을 넘어야 할 경우 페이지 제한 없이 늘리되 최소 장수로 유지한다

**③ 외부 고객 / 협력사**
- 제목: 상대방 이익이 드러나는 명사형 또는 서술형
- 본문: 내부 용어·프로세스 금지, 상대에게 돌아오는 가치 중심
- 용어: 업계 공용어만, 사내 약어·코드명 사용 금지
- 톤: 신뢰·협력 톤, 일방적 주장 대신 근거 제시

**④ 일반 대중 / 커뮤니티**
- 제목: 서술형·대화체 허용 — 호기심·공감을 유도
- 본문: 배경 설명 포함, 전문용어 사용 시 한 줄 풀이 추가
- 용어: 기초 개념부터 설명, 전문용어는 최소화
- 톤: 친근·유쾌, 독자와 같은 눈높이

> 청중 규칙이 명사형 기본값과 충돌 시 **청중 규칙 우선** — Q2 청중 선택이 제목 스타일을 덮어쓴다.

### 구성 예시 출력 형식

```
[확정 구성]
시각 테제: (이 덱의 분위기·질감·에너지를 한 문장으로 — 이후 모든 디자인 결정의 기준점)
덱 핵심 주장: (이 덱이 전달하려는 것을 한 문장으로 — 청중이 자료를 보고 난 뒤 가져가야 할 하나의 생각)
내러티브 흐름: 오프너 → [근거/증거] → [상세/스토리] → 결론/CTA

슬라이드 1  (cover)          : 패키지 디자인 가이드라인 2026 / 부제목
슬라이드 2  (contents)       : 3개 섹션 목차
슬라이드 3  (section-divider): 01 — 디자인 원칙
슬라이드 4  (statistics)     : 색상 / 타이포 / 레이아웃  →  "3가지 원칙이 브랜드 일관성을 만든다"
슬라이드 5  (section-divider): 02 — 소재 사용 규정
슬라이드 6  (content)        : 승인 소재 목록  →  "승인된 소재만 사용하면 교체 비용이 없다"
슬라이드 7  (section-divider): 03 — 적용 사례
슬라이드 8  (content)        : 제품별 적용 예시  →  "실제 사례를 보면 적용 기준이 명확해진다"
슬라이드 9  (closing)        :
```

> 핵심 메시지는 내용 슬라이드(cover·section-divider·closing 제외)에 한 줄씩 명시한다.
> 한 문장으로 표현 안 되면 슬라이드를 분리하거나 목적을 재정의한다.
