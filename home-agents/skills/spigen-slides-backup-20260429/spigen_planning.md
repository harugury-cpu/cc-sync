# spigen_planning.md — 콘텐츠 우선 기획 워크플로우
> 4단계 파이프라인: 소스 분석 → 내러티브 → 슬라이드 아웃라인 → 컴포넌트 매핑 → 렌더

## 핵심 원칙

```txt
1. 내용이 컴포넌트보다 먼저 확정된다
2. 각 단계는 사용자 승인 없이 다음 단계로 넘어가지 않는다
3. Stage 2(아웃라인) 완료 전까지 컴포넌트 이름·코드를 언급하지 않는다
4. 소스 자료가 있으면 AI가 창작하지 않는다 — 추출하고 확인받는다
```

---

## Stage 0: 소스 분석 (자료 있을 때 필수)

### 0-0. 자료 수신 감지

사용자가 아래 중 하나를 제공한 경우 Stage 0을 실행한다:
- 문서·파일 (텍스트, PDF, 보고서, 기획서)
- 코드·시스템 구조 설명
- 옵시디언 노트 / 볼트 파일
- Google Docs·Sheets 링크
- KPI·성과 데이터 (수치, 달성률, 목표)
- 채팅 기록·회의록·이메일

**자료 없이 주제만 주어진 경우**: Stage 0을 건너뛰고 Stage 1로 직행.  
단, 이 경우 `source_mode: "creative"` 로 기록하고 내용 품질 보장이 어렵다는 점을 사용자에게 고지한다.

### 0-1. 핵심 주장 추출

제공된 자료를 읽고 아래 형식으로 출력한다:

```
[소스 분석 결과]

핵심 주장 (1~3개):
1. ___
2. ___
3. ___

뒷받침 근거:
- 수치·데이터: ___
- 사례·증거: ___
- 조건·전제: ___

자료에서 중요도 신호 (발견된 것만):
- 반복 등장하는 개념: ___
- 강조 표현·헤딩 구조: ___
- 명시적 결론·권고: ___

빠진 정보 (발표자가 채워야 할 것):
- ___

추정 목적:
- 이 자료로 청중에게 전달하려는 것: ___
```

### 0-2. 소스 분석 확인 게이트 (필수)

위 결과를 사용자에게 보여주고 아래를 묻는다:

```
"핵심 주장이 맞나요? 빠진 내용이나 다르게 강조하고 싶은 것이 있으면 알려주세요."
```

사용자가 수정·추가하면 내용을 반영하고 다시 확인받는다.  
**사용자 확인 완료 전 Stage 1로 진행 불가.**

## Stage 1: 기획 (Q&A + 내러티브 정의)

### 1-0. 요청 유형 감지 (질문 전 분기)

Q1 답변 또는 최초 요청 문구에서 아래 키워드가 감지되면 1-1 질문 플로우를 건너뛰고 자료 수집 모드로 진입한다.

감지 조건:
- KPI / 성과 / 실적 / 달성률 / 목표 보고 / 상반기 보고 / 연간 보고 / H1 / H2

→ 자료 수집 모드 진입:

```
"KPI·성과 보고서군요. 관련 자료를 공유해주시면 내용을 채워서 드래프트를 만들어드릴게요.
아래 중 편한 방식으로 주세요:

① Google Sheets / Docs 링크
② 기존 PPT / 보고서 링크 또는 캡처
③ 텍스트로 직접 붙여넣기"
```

자료 수신 후 Stage 0 소스 분석 실행 → 확인 게이트 → 아래 값 자동 결정 (묻지 않음):
- 테마: light 고정
- 컴포넌트: `mk_kpi_status_light()` + `mk_kpi_dense_table()` 세트 고정
- Stage 1 Q1~Q4 자동 처리 후 Stage 2로 진행

### 1-1. 준비 질문 수집

Stage 0에서 자료를 분석한 경우: Q1은 확인만, 나머지는 필요한 것만 질문.  
질문은 **하나씩 순서대로** 제시한다. "Decide for me" 선택 시 Claude가 맥락에 맞는 최적 옵션을 결정하고 이유를 한 줄 설명한다.

---

**Q1. 주제**
"어떤 내용의 프레젠테이션인가요?" (자유 입력)  
Stage 0에서 분석한 경우: "소스 분석에서 [___]으로 파악했는데 맞나요?"

**Q2. 청중**
"누가 보는 자료인가요?"
① 내부 팀 (실무 동료)
② 경영진 (의사결정권자)
③ 외부 고객 / 협력사
④ 일반 대중 / 커뮤니티
⑤ 기타 (직접 입력)

**콘텐츠 밀도 원칙 (고정 — 묻지 않음)**
모든 슬라이드는 디테일용 기준으로 작성한다:
- 슬라이드당 본문 글머리 5~6줄 허용
- 수치·근거·세부 단계 생략 없이 전부 기입
- 카드·블록 설명이 자기완결적이어야 함 (발표자 설명 없이 이해 가능)

**Q3. 덱의 목적**
"이 슬라이드로 무엇을 달성하고 싶으신가요?"
① 동기부여 (청중에게 확신 심어주기)
② 사용법 전수 (구체적 실습·방법론)
③ 조직 설득 (도입 제안·승인 요청)
④ 홍보·소개 (커뮤니티·외부 공개)
⑤ 기타 (직접 입력)

**Q4. 테마**
> KPI·성과 보고 유형이면 이 질문을 건너뛴다 — light 고정, 사용자에게 묻지 않음.

"어떤 테마로 만들까요?"
① Dark (기본 / 권장)
② Light

### 1-2. 비주얼 테제 작성

수집한 정보를 바탕으로 **한 문장**으로 무드·방향·에너지를 정의한다.

예시: _"제품의 신뢰성을 데이터로 증명하는, 간결하고 자신감 있는 톤의 덱"_

### 1-3. 내러티브 구조 설계

내지 장수: 내용이 요구하는 만큼 자유롭게 결정한다.  
cover·closing·toc·section-divider는 카운트에서 제외한다.

| 단계 | 역할 | 내지 장수 |
|-----|------|---------|
| **Opener** | 전제·약속·정체성 제시 | cover (카운트 제외) |
| **Problem / Context** | 현황·문제·배경 | 1장 |
| **Solution / Proof** | 핵심 근거·메커니즘·데이터 | 1~2장 |
| **Impact / Detail** | 구체적 결과·수치·기대효과 | 0~1장 |
| **Close / CTA** | 결론·권고·다음 액션 | closing (카운트 제외) |

> **이 단계에서 컴포넌트 이름을 언급하지 않는다.**  
> 형식이 아니라 내용의 논리 흐름만 정의한다.

섹션 구분(section-divider)은 내지 4장 이상일 때만 사용한다.  
섹션 구분이 1개뿐이라면 전체 생략하고 내용 슬라이드로 바로 진입한다.  
closing(요약 페이지): 항상 생략. 내지 마지막 장으로 마무리.

### 1-4. 내러티브 확인 게이트 (필수)

```
"이 논리 흐름이 맞나요? 빠진 단계가 있거나 순서를 바꾸고 싶으면 알려주세요."
```

**사용자 확인 완료 전 Stage 2로 진행 불가.**

## Stage 2: 슬라이드 아웃라인 (내용 확정)

### 2-1. 슬라이드별 핵심 메시지 + 내용 작성

각 슬라이드에 대해 아래를 작성한다. **이 단계에서 컴포넌트 함수명을 절대 언급하지 않는다.**  
사용자가 보는 것은 내용과 구조뿐이어야 한다.

```
[슬라이드 N] (유형) — 레이아웃: (시각적 설명)
핵심 메시지: "___는 ___하다"
제목: (슬라이드에 표시될 실제 제목 텍스트)
구성:
  - (블록명): (들어갈 텍스트 또는 항목 목록)
  - (블록명): (들어갈 텍스트 또는 항목 목록)
  ...
```

> **레이아웃 설명 원칙**: 시각적/기능적 설명만 사용한다.  
> 예: "팀별 역할 행 구분 다이어그램", "Before/After 비교표", "3개 카드 레이아웃", "단계별 흐름도"  
> 함수명은 Stage 3에서만 등장한다.

> **내용 원칙**: Stage 0에서 확인된 핵심 주장이 반드시 반영돼야 한다.  
> source_mode가 "creative"인 경우: 아래 슬롯을 명시적으로 채운다.  
> `actor(누가) / condition(언제·어떤 조건) / action(무엇을) / state_change(어떻게 바뀌는가) / next_action(그 다음)`

### 2-2. 슬라이드 아웃라인 확인 게이트 (필수)

아웃라인을 사용자에게 보여주고:

```
"각 슬라이드의 핵심 메시지와 내용이 맞나요? 수정할 것이 있으면 알려주세요."
```

수정 요청이 있으면 해당 슬라이드만 고치고 다시 제시한다.  
**사용자 확인 완료 전 Stage 3으로 진행 불가.**

### 2-3. 아웃라인 자가검수 (확인 전 실행)

```
□ 각 내지 슬라이드에 "핵심 메시지"가 한 문장으로 작성됐는가?
  → "___는 ___하다" 형식. 없으면 슬라이드 존재 이유가 없다.
□ Stage 0에서 확인된 핵심 주장(1~3개)이 슬라이드 내용에 반영됐는가?
  → source_mode "creative"면 info_slots 5개가 빈칸 없이 채워졌는가?
□ 주장을 뒷받침하는 수치·데이터·근거가 포함됐는가?
  → 없으면 "발표자 보충"이라고 명시하거나 근거를 추가한다.
□ 내러티브가 오프너 → 근거 → 상세 → 결론/CTA 순인가?
□ 밀도 높은 슬라이드가 3장 연속이 아닌가?

청중별 체크 (Q2 기준):
□ 내부 팀: 실행 주체·상태 변화·다음 액션이 각 슬라이드에 명시됐는가?
□ 경영진: 수치·임팩트·결론이 제목과 첫 블록에 먼저 보이는가?
□ 외부: 내부 용어가 제거됐는가? 상대방 가치가 명시됐는가?
```

## Stage 3: 컴포넌트 매핑 (자동 실행)

사용자 승인 완료 후 자동으로 실행한다. 사용자에게 별도 확인을 요청하지 않는다.  
이 단계부터 컴포넌트 이름이 등장한다.

### 3-0. 청중 행동 유형 판정 (audience_action)

컴포넌트 선택 전에 **이 슬라이드에서 청중이 무엇을 해야 하는지** 결정한다.

```txt
understand — 내용을 이해/파악하면 됨 (기본값)
decide     — 두 경로 중 하나를 직접 선택/판단해야 함 → mk_decision_tree() 우선
execute    — 지시를 따라 실행해야 함 → mk_flow_focus() 또는 mk_split_cards() 우선
recall     — 나중에 참조하기 위한 레퍼런스 자료
```

"비교처럼 생긴 내용"이더라도 `audience_action: decide`이면 `mk_compare_rows`가 아니라 `mk_decision_tree`다.

### 3-1. 컴포넌트 선택 규칙

`spigen_render_rules.md` 섹션 8의 결정 테이블을 참조한다.

**내용 우선 원칙**:
1. 슬라이드의 항목 수·줄 수·텍스트 길이를 먼저 파악한다
2. 그 내용을 온전히 담을 수 있는 컴포넌트를 선택한다
3. 내용이 컴포넌트 한도를 초과하면 → 더 큰 컴포넌트로 교체하거나 페이지를 분리한다

내용을 줄이거나 잘라서 레이아웃에 끼워맞추지 않는다.

**사용 가능한 컴포넌트 전체 목록**:

| 컴포넌트 | 언제 쓰나 | 가이드 시트 |
|----------|---------|------------|
| `mk_compare_rows` | Before/After 비교 · 도입 전후 | p.21 |
| `mk_flow_focus` | 단계별 프로세스 · 청중이 실행할 때 | p.22 |
| `mk_flow` | 수평 프로세스 + 비용/세부정보 | p.23 |
| `mk_3col_cards` | 독립 항목 3개 병렬 | p.24 |
| `mk_swimlane_mapping` | 팀·역할별 행 구분 · 다자 협업 | p.25 |
| `mk_decision_tree` | Yes/No 분기 판단 | p.26 |
| `mk_split_cards` | 텍스트 설명 + 상태카드 4개 | p.27 |
| `mk_arch_layers` | 시스템 레이어 구조 | p.28 |
| `mk_arch_orchestrator` | 모듈 관계·호출 구조 | p.29 |
| `mk_rule_grid` | 규칙·가이드 카드 2열 | p.30 |
| `mk_split` | 좌우 2분할 비교 | p.31 |
| `mk_text_block` | 긴 본문 텍스트 · 세부 설명 | p.32 |
| `mk_callout_message` | 핵심 한 줄 강조 메시지 | p.33 |

### 3-2. 입력 자산 충분성 판정

```txt
Q. 이 자료가 실제 UI / 보드 / 칼럼 / 버튼 / 위치를 설명해야 하는가?
```

- **No** → 기존 방식대로 진행
- **Yes + 스크린샷 있음** → 캡처를 근거 자산으로 사용해 상세 설명 진행
- **Yes + 스크린샷 없음** → 실제 위치 단정 금지, **개념 흐름 자료**로 범위를 낮춘다

슬라이드별로 생성한다:

### 3-4. 컴포넌트 재검토 허용 조건

기본적으로 audience_action과 결정 테이블에서 선택된 컴포넌트를 유지한다.  
아래 조건 중 하나라도 해당하면 재검토를 허용한다 (단순 변심은 금지):

```txt
1. info_slots 2개 이상이 선택된 컴포넌트로 표현 불가능한 경우
   예: mk_flow_focus() 선택 → condition 분기가 핵심인 슬라이드
       → mk_decision_tree()로 재검토 허용

2. 실제 항목 수가 컴포넌트 적정 범위와 2개 이상 차이 날 때
   예: 항목 5개인데 mk_3col_cards() 선택
       → mk_split_cards() 또는 mk_5col_steps() 재검토 허용
```

재검토 시 반드시:
- COMPONENT_BRIEF의 function_name, rationale을 업데이트
- 변경 이유를 한 줄로 명시 (어느 조건에 해당했는지)

오버플로 처리 우선순위:
```txt
1) 텍스트 압축
2) 같은 컴포넌트 안에서 줄 수 축소
3) 슬라이드 분리
→ "렌더링이 어려워 보여서" 다른 컴포넌트로 바꾸는 것은 금지
```

### 3-5. ckm-slides 패턴 추천기 사용 원칙

역할: 이 슬라이드가 comparison / metrics / timeline / feature-grid 중 무엇에 가까운지 레이아웃 패턴 이름 후보를 추천.

제한: HTML 슬라이드 생성 결과를 그대로 채택하지 않는다. 패턴 이름과 정보 구조만 참고한다.

## 카피 작성 원칙

- **슬라이드당 하나의 메시지**: 경쟁하는 블록 금지
- **3~5초 스캔 가능하게**: 본문 글머리는 3줄 이내
- **제목 스타일**: 항상 명사형 ('AI 활용 원칙', '비용 절감 사례')
- **AI 클리셰 금지**: 혁신적인·원활한·극대화·시너지·솔루션·최적화·스마트한·강력한·게임체인저 사용 금지 → 구체적 수치·사실로 대체
- **코드·함수명 금지**: 슬라이드 텍스트에 `mk_flow()`, `spigen_lib.py` 같은 함수명 직접 기재 금지
- **불필요한 선·여백 금지**: 장식용 구분선, 빈 사각형, 투명 여백 도형 추가 금지
- **여백 우선**: 공간을 채우려 하지 말 것

## 덱 내러티브 리듬

- **오프너 → 근거 → 상세 → 결론/CTA**: 덱 전체를 이 흐름으로 구성한다
- **밀도 교차**: 정보 밀도 높은 슬라이드 뒤에는 시각 템포를 리셋하는 단순 슬라이드를 배치
- **섹션 구분이 템포 리셋**: 섹션 구분 없이 밀도 높은 슬라이드를 연속 3장 이상 쌓지 않는다
- **도입부는 단순하게**: cover는 제목·전제·약속 하나만 담는다

## 청중별 카피 규칙 (Q2 반영)

**① 내부 팀 (실무 동료)**
- 제목: 명사형 또는 서술형 모두 허용 — 팀 내 통용 약어·용어 그대로 사용
- 본문: 현황·프로세스·수치 중심, 3줄 이내 밀도 허용
- 톤: 간결·건조, 배경 설명 최소화

**② 경영진 (의사결정권자)**
- 제목: 명사형 고정
- 본문: 수치·임팩트·결론 먼저, 세부 내용도 슬라이드에 전부 기입
- 톤: 간결·자신감, "~합니다" 확정형 선호
- **섹션 구분 없음**: section-divider 사용 안 함
- **페이지 압축 우선**: 4장 이내에 최대한 밀도 있게

**③ 외부 고객 / 협력사**
- 제목: 상대방 이익이 드러나는 명사형 또는 서술형
- 본문: 내부 용어·프로세스 금지, 상대에게 돌아오는 가치 중심
- 톤: 신뢰·협력 톤, 일방적 주장 대신 근거 제시

**④ 일반 대중 / 커뮤니티**
- 제목: 서술형·대화체 허용 — 호기심·공감을 유도
- 본문: 배경 설명 포함, 전문용어 사용 시 한 줄 풀이 추가
- 톤: 친근·유쾌, 독자와 같은 눈높이

> 청중 규칙이 명사형 기본값과 충돌 시 **청중 규칙 우선**.

## 구성 예시 출력 형식

```
[확정 구성]
시각 테제: (이 덱의 분위기·질감·에너지를 한 문장으로)

내러티브 흐름: 오프너 → [근거/증거] → [상세/스토리] → 결론/CTA

슬라이드 1  (cover)          : [제목]
슬라이드 2  (section-divider): 01 — [섹션명]
슬라이드 3  (content)        : [제목]  →  "[핵심 메시지]"

...
```

> 핵심 메시지는 내용 슬라이드(cover·section-divider·closing 제외)에 한 줄씩 명시한다.
